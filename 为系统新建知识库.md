# 为新系统构建 Agent 可加载的知识库

这是一个非常有价值的问题。核心挑战是：**如何把一个系统的"全部上下文"压缩成 agent 可以高效加载和使用的结构化知识**。下面我按照"从零开始设计"的思路来讲。

## 一、先想清楚：知识库到底要装什么

一个系统的完整上下文，本质上分为四层：

**事实层**（What）—— 系统有什么。包括领域模型、数据结构、API 接口、模块清单、外部依赖、配置项。这层是静态的、可枚举的。

**规则层**（How）—— 系统怎么运作。包括业务规则、状态机、工作流、权限模型、校验逻辑、错误处理约定。这层是行为性的。

**意图层**（Why）—— 为什么这样设计。包括架构决策（ADR）、取舍记录、废弃方案、历史演进。这层最容易缺失，但对 agent 做判断最关键——没有它，agent 会不断重复已经被否决的方案。

**操作层**（Do）—— 怎么动手。包括本地启动步骤、常用命令、调试技巧、发布流程、常见故障排查。这层是 runbook。

很多团队只做了事实层（等同于 API 文档），结果 agent 拿到后只能回答"是什么"，无法回答"该不该这么改"。

## 二、采用"渐进披露"架构，而不是"全量灌入"

新系统最容易犯的错是把所有文档塞进一个 5 万字的 `CONTEXT.md`，然后让 agent 每次加载。这样做 token 成本爆炸，而且 agent 的注意力会被稀释。

更好的做法是**分层加载**：

```
knowledge-base/
├── INDEX.md              ← agent 必读，只有这个是默认加载的
├── overview/
│   ├── system-purpose.md
│   ├── glossary.md       ← 术语表，消除歧义
│   └── architecture.md
├── domains/              ← 按业务域切分
│   ├── order/
│   │   ├── model.md
│   │   ├── rules.md
│   │   └── api.md
│   └── payment/
├── decisions/            ← ADR，每个决策一个文件
│   ├── 0001-why-event-driven.md
│   └── 0002-why-not-graphql.md
├── runbooks/
└── skills/               ← 可执行的操作指南
```

`INDEX.md` 是入口，它**不包含内容，只包含路由**：描述每个子文件里有什么、什么情况下该读。agent 先读 INDEX，然后按需加载具体文件。这和人类看书先翻目录是一个道理。

## 三、每个文档的写法要"为 agent 优化"

给 agent 看的文档和给人看的不一样。几个关键原则：

**前置结论**。每个文件开头用 2-3 句说清楚"这个文件回答什么问题、不回答什么问题"。agent 扫一眼就能决定要不要继续读。

**显式化隐性知识**。人类工程师知道"订单金额永远不能为负"是常识，但对 agent 必须写出来。凡是代码里用 assert、用类型约束、用 if 判断的条件，都应该在知识库里有对应的自然语言声明。

**负面知识同样重要**。写清楚"我们试过 X，因为 Y 原因放弃了"。这比只写"我们用 Z"有用得多，能防止 agent 提议回到旧方案。

**代码和文档共置**。如果用 Markdown，在文档里嵌入真实的代码片段（不是伪代码），并标注来源文件路径。agent 能据此交叉验证。

## 四、关键设计：可执行的 Skill 而不是纯文档

纯文档告诉 agent "知道什么"，但 agent 真正需要的是"该怎么做"。参考 Anthropic 在 Claude 的 skill 设计思路，每个高频操作应该做成一个独立的 `SKILL.md`，里面包含：

触发条件（什么情况下用这个 skill）、前置检查、执行步骤、验证方法、常见错误。

比如"创建一个新的订单类型"这种任务，不应该让 agent 自己从架构文档里推断，而应该有一个 `skills/add-order-type/SKILL.md`，明确写出要改哪几个文件、要加什么测试、要更新哪些配置。

## 五、让知识库"活着"的机制

知识库最大的风险是**和代码脱节**。新系统就要建立防腐机制：

**代码是唯一真相源的部分自动生成**。API 文档、数据库 schema、配置项清单这些应该从代码自动抽取，而不是手写。可以用 OpenAPI、JSON Schema、TypeScript 类型定义作为中间层，构建时生成对应的 Markdown。

**决策和意图由人维护**。ADR、业务规则、架构演进这些必须人工写，但要强制和 PR 绑定——修改核心模块必须同时提交 ADR 更新，否则 CI 失败。

**加一个"新鲜度"元数据**。每个文档头部标注 `last_verified: 2026-04-15` 和 `applies_to_version: >=2.3.0`。agent 加载时能判断信息是否可能过期。

**定期让 agent 自检**。用 agent 读知识库，然后让它描述系统，再和真实代码行为对比。不一致的地方就是知识库的缺口。这个过程可以自动化。

**不要追求一次性完备**。知识库应该和系统一起长大。关键是建立起结构和维护规范，让每次写代码都伴随着知识库的小量更新，而不是攒到最后集中补文档——那样一定会失败。
